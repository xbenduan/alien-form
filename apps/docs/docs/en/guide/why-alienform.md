# Why AlienForm: A Governance-first Schema Form Engine

AlienForm is not a Formily clone, and it does not try to reimplement an existing solution with a different API surface. It focuses on a more specific problem: when form schemas keep growing, are increasingly generated with AI assistance, and need to be maintained inside enterprise systems for years, how should a Schema Form engine keep the protocol clear, the boundaries stable, and the runtime behavior auditable?

Formily is a mature Schema Form solution with a rich model, protocol, and ecosystem. AlienForm learns from proven ideas such as schema-driven forms, reactive field models, and layered architecture, but it deliberately takes a more constrained path. Instead of becoming the most feature-complete form platform, AlienForm prioritizes being a governance-first protocol layer for enterprise forms and human-AI collaborative development.

## Positioning

> AlienForm is a governance-first Schema Form engine for enterprise forms that are written, reviewed, debugged, and maintained by both humans and AI.

Governance-first is not an abstract slogan. It shows up in several concrete design choices:

- Keep the protocol constrained and reduce implicit behavior.
- Keep a field's behavior as local and explainable as possible.
- Move async work, side effects, authentication, and caching into external handlers.
- Let core focus on form state, field trees, reactions, formatting, validation, and protocol execution.
- Make each schema fragment readable by humans and easier for AI to generate and repair correctly.

## Why Not a Formily Clone

AlienForm does not aim to become “another Formily” because Formily already exists and is mature in the areas it focuses on. Rebuilding a smaller version of the same thing would not create much new value.

AlienForm focuses on a different set of questions:

- When schemas are continuously generated by multiple teams or AI tools, how do we prevent protocol sprawl?
- When form behavior goes wrong, how do we identify which rule affected which field?
- When an enterprise needs to review dynamic form configuration, how do we keep the configuration readable and explainable?
- When business logic needs remote data, permissions, caching, and API calls, how do we keep core from being polluted by business semantics?

So the relationship between AlienForm and Formily is not replacement or imitation. They operate in the same problem space, but with different product priorities.

| Concern               | Mature general-purpose Schema Form                              | AlienForm's choice                                                        |
| --------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------------- |
| Goal                  | Cover a broad range of form modeling and rendering needs        | Prioritize protocol clarity, auditability, and governance boundaries      |
| Protocol design       | Provide rich expressive power                                   | Reduce concept count and control dynamic entry points                     |
| Business side effects | Can be integrated through multiple mechanisms                   | Explicitly route them through registered business handlers                |
| AI collaboration      | AI can generate configuration, but extra constraints are needed | Lower generation cost through constrained and explainable protocol design |
| Enterprise governance | Relies on team conventions and engineering practices            | Moves governance concerns into protocol and runtime boundaries            |

This is not a value judgment. It is a different set of trade-offs for a different set of scenarios. AlienForm aims to give enterprise developers lower cognitive load and higher maintenance certainty in form-heavy systems.

## Why the Protocol Should Be Constrained

The easiest way for a Schema Form system to become unmaintainable is not field count. It is overly flexible dynamic behavior.

When a protocol allows conditions, actions, target field mutations, side effects, remote requests, and script execution to be mixed together, it looks powerful at first. Over time, however, it becomes harder to answer basic questions:

- Why is this field hidden?
- Who loaded this option list?
- Why was this value rewritten?
- Why does this validation only run in some cases?
- What will be affected if I change this field?

AlienForm's choice is to keep the protocol as flat and local as possible. For example, `x-reaction` describes how a property of the current field is derived, rather than acting as a command system that mutates other fields.

That means schema should look more like this:

```ts
{
  name: 'city',
  title: 'City',
  'x-reaction': {
    target: 'dataSource',
    rule: {
      type: 'computed',
      handler: 'loadCitiesByProvince'
    }
  }
}
```

Instead of spreading imperative cross-field actions across the schema.

This constraint brings direct benefits:

1. **Easier to read**: most behavior can be understood locally from the field definition.
2. **Easier to review**: dynamic properties are explicit and traceable.
3. **Easier to debug**: issues can be followed through field properties and rule execution.
4. **Easier to generate**: AI has fewer valid shapes to choose from.
5. **Easier to maintain**: the protocol does not keep expanding into a platform language.

Constraint does not mean reduced capability. Complex business logic is still possible, but complexity should live in the right layer.

## Why Async Side Effects Do Not Belong in Schema

Enterprise forms often need async capabilities, such as:

- Load members after a department is selected.
- Query a price by product code.
- Validate whether a username already exists.
- Decide whether a field is editable based on permissions.
- Run a business pre-check before submit.

These capabilities are related to forms, but they are not core form engine semantics. They involve API contracts, authentication, caching, error handling, retry policies, environments, logging, and permission models. Different enterprises and business units implement them differently.

If those capabilities are built directly into schema or core, the short-term experience may feel convenient, but the long-term cost is high:

1. **Core becomes heavy**: the form engine gradually turns into a business platform layer.
2. **The protocol becomes complex**: schema needs to describe URLs, methods, headers, caching, and error behavior.
3. **Security boundaries become blurry**: review becomes harder when configuration contains request and authentication semantics.
4. **Testing becomes harder**: form state tests become polluted by remote dependencies.
5. **Migration becomes painful**: API strategy changes start to affect the schema protocol itself.

AlienForm keeps this boundary explicit: core defines when a handler is called, what context it receives, and what result it should return. Business code injects the actual implementation through `handlers`.

```ts
createForm({
  handlers: {
    async loadCitiesByProvince(ctx) {
      return requestCities(ctx.values.province);
    },
  },
});
```

The schema references the handler by name, without embedding request details:

```ts
{
  name: 'city',
  'x-reaction': {
    target: 'dataSource',
    rule: {
      type: 'computed',
      handler: 'loadCitiesByProvince'
    }
  }
}
```

This keeps core stable while preserving enough freedom for business code. More importantly, it keeps schema as a reviewable protocol rather than a script that hides network behavior.

## Why Auditability Matters

For simple forms, rendering correctly may be enough. In enterprise systems, dynamic forms often sit at the entry point of approval, permission, contract, finance, HR, and compliance workflows. A form is not just UI; it is a carrier of business rules.

An enterprise schema should be able to answer:

- Which fields are controlled by permissions?
- Which fields are derived from other fields?
- Which rules trigger async handlers?
- Which validations can block submission?
- Which values are formatted before use?
- Why did a user action change a field's state?

This is why AlienForm emphasizes auditability. Dynamic behavior should not only exist inside a runtime black box. It should be readable, traceable, explainable, and testable.

AlienForm's auditability direction includes:

- **Readable protocol**: dynamic rules are stable, local, and explicit.
- **Traceable execution**: reactions, formatting, and validation can be recorded and explained.
- **Analyzable dependencies**: field dependencies can be extracted into graphs or reports.
- **Actionable errors**: invalid rules, missing handlers, and wrong return values can produce clear diagnostics.
- **Testable behavior**: core logic can be tested without UI.

These capabilities are not only useful to framework authors. They directly determine whether an enterprise team can trust a schema to be maintained by multiple developers, or even generated with AI assistance.

## Why It Fits AI Generation Better

AI is good at generating structured configuration from requirements, but it also tends to make predictable mistakes:

- Generate component names or handler names that do not exist.
- Put async logic into a chain that should remain synchronous.
- Mix too much imperative logic into schema.
- Introduce an overly complex structure to solve a local problem.
- Produce code that works once but is hard to maintain.

The freer the protocol, the larger the generation space and the larger the error space. AlienForm reduces the number of valid shapes so AI has fewer ways to be wrong.

This shows up in several ways:

1. **Few rule types**: `static`, `expression`, `match`, and `computed` cover most derivation scenarios.
2. **Clear responsibility boundaries**: formatting, reactions, validation, and business side effects each have a specific place.
3. **Explicit handler references**: AI can reference business capabilities without inventing request details.
4. **More lintable schema**: stable structure makes static checks and automated repair suggestions easier.
5. **Reusable examples**: AI can learn from recipes instead of improvising freely.

So AI-friendly does not mean “put an LLM inside the form engine”. It means the protocol itself is easier for AI to generate, easier for humans to review, and easier for tools to validate.

## Why It Fits Enterprise Governance

Enterprise governance is not about slowing developers down. It is about keeping the system controllable as it scales.

In form-heavy systems, governance usually means:

- New developers can understand existing schemas.
- Teams can estimate the impact of business rule changes.
- Configuration can be checked automatically before release.
- Production issues can be replayed and diagnosed.
- Different teams can share the protocol while keeping their own business implementations.
- AI-generated configuration still has tools and rules as guardrails.

AlienForm's layered design supports these needs:

- `@alien-form/core` stays headless and focuses on model and protocol execution.
- `@alien-form/react` connects core to React.
- `@alien-form/ui` is only a default UI implementation, not a business component system.
- `handlers` act as business extension points for remote data and side effects.
- schema remains a readable, validatable, and explainable protocol layer.

This allows enterprises to integrate their own permissions, APIs, telemetry, caching, and component libraries without modifying core. It also allows schema to be reviewed without exposing business implementation details.

## Design Principles

AlienForm can be summarized by five principles:

### 1. Core should stay boring

Core should be stable, constrained, and testable. It should not know where business APIs live or how an enterprise authenticates requests. The more boring core is, the more reliable it becomes.

### 2. Schema should be readable before powerful

Schema should first be readable by humans, then expressive enough for complex cases. Complex capability can be extended through handlers, but the protocol itself should not become another hard-to-govern language.

### 3. Async belongs to business handlers

Async work and side effects belong to the business layer. The form engine defines when to call, what context to pass, and how to consume the result.

### 4. Reactions should explain field properties

Reactions should explain how field properties are derived, rather than becoming a cross-field imperative action system.

### 5. AI output must be reviewable

AI-generated schema must be reviewable by humans, checkable by tools, and traceable at runtime. Otherwise, AI only accelerates the production of unmaintainable configuration.

## Suitable and Non-goals

AlienForm is suitable for:

- Enterprise admin forms.
- Large numbers of dynamic forms that need long-term maintenance.
- Projects where multiple developers collaborate on schema.
- Projects that use AI to generate or migrate form configuration.
- Projects that need clear separation between core, UI, and business side effects.
- Projects that need to debug, audit, and test form protocols.

AlienForm does not primarily aim to:

- Become the largest low-code platform.
- Build every remote request and business data protocol into core.
- Use schema to express arbitrary imperative business workflows.
- Cover the full material ecosystem of every UI component library.
- Copy every capability of mature frameworks in the early stage.

These are not permanent limitations. They are product-stage and design-priority choices.

## Summary

AlienForm does not exist to prove that “we can also build a Formily”. It exists to answer a newer question: when AI participates more deeply in development, when enterprise configuration becomes more complex, and when dynamic forms require stronger governance, how should a Schema Form engine be designed?

AlienForm's answer is:

- More constrained protocol.
- Clearer boundaries.
- Externalized side effects.
- More traceable behavior.
- Schema that is easier for AI to generate and easier for humans to review.

If you need a governance-first, human-AI friendly, long-term maintainable protocol layer for enterprise forms, that is the problem AlienForm is built to solve.
