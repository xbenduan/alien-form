# Why AlienForm: A Governance-first Schema Form Engine

AlienForm 不是 Formily 的克隆，也不是试图用另一套 API 重新实现一遍已有能力。它面向的是一个更具体的问题：当表单配置越来越多、越来越依赖 AI 生成、越来越需要在企业内长期维护时，Schema Form 引擎应该如何让协议更清晰、边界更稳定、行为更可审计。

Formily 是非常成熟的 Schema Form 方案，拥有完整的模型、协议和生态。AlienForm 借鉴了 Schema 驱动、响应式字段模型和分层设计这些成熟思想，但它选择了一条更收敛的路线：不追求成为能力最全的表单平台，而是优先成为一个适合企业治理和人机协同开发的表单协议层。

## 一句话定位

> AlienForm 是一个治理优先的 Schema Form 引擎，面向人和 AI 共同编写、审查、调试和长期维护企业表单的场景。

这里的“治理优先”不是一个抽象口号，而是贯穿在几个设计取舍里：

- 协议能力保持收敛，减少隐式行为。
- 字段自己的行为尽量在字段本地闭环表达。
- 异步、副作用、鉴权、缓存等业务语义交给外部 handlers。
- core 只负责表单状态、字段树、联动、格式化、校验和协议执行。
- 每一段 schema 都应该尽量容易被人读懂，也容易被 AI 正确生成和修正。

## 为什么不是 Formily Clone

AlienForm 不想成为“另一个 Formily”，原因很简单：Formily 已经存在，而且在它擅长的领域足够成熟。重复实现一个低配版本并不能创造新的价值。

AlienForm 更关注的是另一类问题：

- 当 schema 是由多人、多个业务线、甚至 AI 持续生成时，如何避免协议失控？
- 当表单逻辑出现问题时，如何快速判断是谁影响了哪个字段？
- 当企业需要审查动态表单配置时，如何让配置本身足够可读、可解释？
- 当业务需要接入远程数据、权限、缓存、接口请求时，如何避免 core 被业务语义污染？

因此，AlienForm 与 Formily 的关系不是“替代”和“复制”，而是站在 Schema Form 这个问题域里，选择了不同的产品重心。

| 关注点     | 成熟通用型 Schema Form          | AlienForm 的选择                         |
| ---------- | ------------------------------- | ---------------------------------------- |
| 目标       | 覆盖广泛表单建模和渲染需求      | 优先保证协议清晰、行为可审计、边界可治理 |
| 协议设计   | 倾向提供丰富表达能力            | 倾向减少概念数量，控制动态能力入口       |
| 业务副作用 | 可以通过多种机制接入            | 明确收口到业务注册的 handlers            |
| AI 协同    | AI 可以生成配置，但需要额外约束 | 从协议收敛和可解释性上降低 AI 生成成本   |
| 企业治理   | 依赖团队规范和工程实践          | 将治理诉求前置到协议和运行时边界中       |

这不是优劣判断，而是不同阶段、不同场景下的取舍。AlienForm 的目标是让开发者在企业表单这个具体问题上获得更低的理解成本和更高的维护确定性。

## 为什么协议要收敛

Schema Form 最容易失控的地方，不是字段太多，而是动态能力太自由。

当协议同时允许“条件判断”“动作执行”“目标字段修改”“副作用调用”“远程请求”“脚本执行”混在一起时，schema 表面上很灵活，实际维护时会变得越来越难回答几个基本问题：

- 这个字段为什么隐藏了？
- 这个选项列表是谁加载的？
- 这个值为什么被改写了？
- 这个校验为什么在某些情况下才触发？
- 改一个字段会影响哪些地方？

AlienForm 的选择是让协议尽量单层化、局部化。以 `x-reaction` 为例，它描述的是“当前字段的某个属性如何被派生”，而不是“当条件成立时去修改另一个字段”。

这意味着 schema 更像这样：

```ts
{
  name: 'city',
  title: '城市',
  'x-reaction': {
    target: 'dataSource',
    rule: {
      type: 'computed',
      handler: 'loadCitiesByProvince'
    }
  }
}
```

而不是把跨字段命令式动作散落在多个地方。

这种收敛带来的直接收益是：

1. **更容易阅读**：读到一个字段时，可以在本地理解它的大部分行为。
2. **更容易审查**：审查 schema 时能明确看到动态属性来自哪里。
3. **更容易调试**：定位问题时可以沿着字段属性和 rule 链路排查。
4. **更适合生成**：AI 更容易生成结构稳定、边界明确的配置。
5. **更适合长期维护**：协议不因为短期需求不断膨胀成平台语言。

收敛不是削弱能力，而是减少不必要的自由度。复杂业务仍然可以做，但复杂性应该放在更合适的位置。

## 为什么不把异步副作用塞进 Schema

企业表单里经常需要异步能力，例如：

- 根据部门加载成员列表。
- 根据商品编码查询价格。
- 校验用户名是否重复。
- 根据权限决定字段是否可编辑。
- 提交前调用业务接口做预校验。

这些能力看起来都和表单有关，但它们并不属于表单引擎的核心语义。它们涉及接口协议、鉴权、缓存、错误处理、重试策略、灰度环境、日志和权限模型。不同企业、不同业务线的实现方式差异非常大。

如果把这些能力直接放进 schema 或 core，短期会显得方便，长期会带来几个问题：

1. **core 会变重**：表单引擎会逐渐变成业务平台层。
2. **协议会变复杂**：schema 需要描述 URL、method、headers、缓存、错误提示等细节。
3. **安全边界会模糊**：配置里混入请求和鉴权语义后，审查成本显著上升。
4. **测试会变困难**：表单状态测试会被远程依赖污染。
5. **迁移会变痛苦**：接口策略变化会反向影响 schema 协议。

AlienForm 的做法是：core 定义调用时机、上下文和返回值约定，业务通过 `handlers` 注入具体实现。

```ts
createForm({
  handlers: {
    async loadCitiesByProvince(ctx) {
      return requestCities(ctx.values.province);
    },
  },
});
```

schema 只引用 handler 名称，而不直接关心请求细节：

```ts
{
  name: 'city',
  'x-reaction': {
    target: 'dataSource',
    rule: {
      type: 'computed',
      handler: 'loadCitiesByProvince'
    }
  }
}
```

这条边界让 core 保持稳定，也让业务拥有充分的自由度。更重要的是，它让 schema 成为一种可审查的协议，而不是一段隐藏了网络行为的脚本。

## 为什么强调 Auditability

在简单表单里，“能渲染出来”就够了。但在企业系统里，动态表单经常会承载审批、权限、合同、财务、人事和合规流程。此时，表单不只是 UI，而是业务规则的入口。

一份企业级 schema 应该能回答：

- 哪些字段受权限控制？
- 哪些字段由其他字段派生？
- 哪些规则会触发异步 handler？
- 哪些校验会阻止提交？
- 哪些值在提交前会被格式化？
- 某次用户操作为什么导致字段状态变化？

这就是 AlienForm 强调 auditability 的原因。可审计性意味着：动态行为不应该只存在于运行时黑盒里，而应该能被读取、追踪、解释和测试。

AlienForm 的可审计方向包括：

- **协议可读**：schema 中的动态规则尽量稳定、局部、显式。
- **执行可追踪**：reaction、format、validate 的执行链路可以被记录和解释。
- **依赖可分析**：字段之间的依赖关系可以被提取成图或报告。
- **错误可定位**：非法 rule、缺失 handler、错误返回值能给出明确提示。
- **结果可测试**：核心行为可以脱离 UI，以纯模型方式测试。

这些能力不只是给框架作者看的。它们直接影响企业开发者是否敢把一个表单协议交给多人维护，甚至交给 AI 辅助生成。

## 为什么更适合 AI 生成

AI 很擅长根据需求生成结构化配置，但 AI 也容易犯几类错误：

- 生成不存在的组件名或 handler 名。
- 把异步逻辑写进不该异步的链路。
- 在 schema 中混入过多命令式逻辑。
- 为了满足一个场景，引入过度复杂的协议结构。
- 生成能跑但难以维护的“临时代码”。

协议越自由，AI 生成空间越大，错误空间也越大。AlienForm 通过收敛协议减少可选路径，让 AI 更容易生成符合预期的 schema。

这体现在几个方面：

1. **规则类型少**：`static`、`expression`、`match`、`computed` 已经覆盖大部分派生场景。
2. **责任边界清楚**：同步格式化、属性联动、校验、业务副作用各自有明确位置。
3. **handler 名称显式**：AI 可以引用业务能力，但不需要编造请求细节。
4. **schema 更容易 lint**：结构越稳定，越容易做静态检查和自动修复建议。
5. **示例更容易复用**：AI 可以从 recipes 中学习固定写法，而不是自由发挥。

因此，AlienForm 的 AI-friendly 不是“在表单里接一个大模型”，而是让协议本身更适合被 AI 生成、被人审查、被工具校验。

## 为什么更适合企业治理

企业治理的核心不是限制开发者，而是让系统在规模变大后仍然可控。

在表单场景里，治理通常意味着：

- 新人能读懂已有 schema。
- 业务规则变化时能判断影响范围。
- 配置上线前能做自动检查。
- 出现线上问题时能回放和定位。
- 不同团队能共享协议，但保留各自业务实现。
- AI 参与生成后，仍然有工具和规范兜底。

AlienForm 的分层设计正是为了支持这些诉求：

- `@alien-form/core` 保持无头，专注模型和协议执行。
- `@alien-form/react` 只负责把 core 接入 React。
- `@alien-form/ui` 只是默认 UI 实现，不绑定业务组件体系。
- `handlers` 作为业务扩展点，承接远程数据和副作用。
- schema 作为协议层，保持可读、可校验、可解释。

这种分层让企业可以在不修改 core 的情况下接入自己的权限、接口、埋点、缓存和组件库，也可以在不暴露业务实现的情况下审查 schema 本身。

## 设计原则

AlienForm 的设计可以总结为五条原则：

### 1. Core should stay boring

core 应该稳定、克制、可测试。它不应该知道业务接口在哪里，也不应该知道企业如何鉴权。core 越无聊，越可靠。

### 2. Schema should be readable before powerful

schema 首先应该能被人读懂，然后才是表达复杂能力。复杂能力可以通过 handlers 扩展，但协议本身不应该变成另一门难以治理的语言。

### 3. Async belongs to business handlers

异步和副作用属于业务层。表单引擎只定义何时调用、用什么上下文调用、如何接收结果。

### 4. Reactions should explain field properties

联动应该解释字段属性是如何得到的，而不是成为跨字段命令式动作系统。

### 5. AI output must be reviewable

AI 生成的 schema 必须能被人审查、被工具检查、被运行时追踪。否则 AI 只是在加速制造不可维护配置。

## 适合和不适合的场景

AlienForm 更适合：

- 企业中后台表单。
- 需要长期维护的大量动态表单。
- 需要多人协作维护 schema 的项目。
- 需要 AI 辅助生成或迁移表单配置的项目。
- 需要清晰隔离 core、UI、业务副作用的项目。
- 需要调试、审计、测试表单协议的项目。

AlienForm 不优先追求：

- 成为最大而全的低代码平台。
- 内置所有远程请求和业务数据协议。
- 用 schema 表达任意命令式业务流程。
- 覆盖所有 UI 组件库的完整物料生态。
- 在早期阶段复制成熟框架的全部能力。

这不是能力边界的永久限制，而是产品阶段和设计重心的选择。

## 总结

AlienForm 的目标不是证明“我们也能做一个 Formily”。它更想回答一个新的问题：在 AI 参与越来越多、企业配置越来越复杂、动态表单越来越需要治理的时代，Schema Form 引擎应该如何设计？

AlienForm 给出的答案是：

- 协议更收敛。
- 边界更清晰。
- 副作用更外置。
- 行为更可追踪。
- schema 更适合被 AI 生成，也更适合被人审查。

如果你需要的是一个治理优先、人机协同友好、可以长期维护的企业表单协议层，那么这就是 AlienForm 想解决的问题。
