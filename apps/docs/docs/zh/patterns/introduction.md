# 开发范式

本节整理的是 AlienForm 在常见企业表单场景下的推荐写法。

这些页面不是“可任选的几种实现”，而是把已经反复出现的问题收敛成统一模式，帮助团队在以下方面保持一致：

- 逻辑归位清晰
- schema、`setup`、React、decorator、handler 的职责边界清晰
- 避免在 React 层堆积补丁式 `useEffect` 和条件分支
- 让多人协作、代码审查和长期维护更稳定

## 如何阅读这一节

推荐顺序是：

1. 先读 `Guide`，建立 `core / react / ui` 分层和 `setup + form.effect(...)` 的基本心智。
2. 再按具体业务场景查阅对应 pattern。
3. 需要查方法签名或运行时行为时，再回到 `API` 页面。

## 这些 pattern 解决什么问题

Patterns 主要处理的是“业务写法应该如何落位”，例如：

- 编辑态初始化
- 形态切换
- 权限与可见性
- 异步数据
- 复杂组件建模
- 规格与 SKU 销售矩阵

这些页面会统一回答几件事：

- 场景是什么
- 非推荐写法是什么
- 推荐写法应该放在哪一层
- 为什么这套写法更符合当前 runtime 范式

## 一句话原则

先判断逻辑属于 schema、`setup`、decorator 还是业务 handler，再选择具体实现方式。
